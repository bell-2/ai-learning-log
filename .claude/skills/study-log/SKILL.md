# study-log — 학습 노트 적립

학습 노트: 이 저장소의 `ko/study-notes.md`
학습 루프(Karpathy): 실전에서 걸린 문제를 on-demand로 파고 직접 확인(1) → 노트 적립, 블로그에 같은 내용 발행(2) → 정리 세션마다 과거의 나와 비교 회고(3)

> 이 저장소를 클론해서 자기 학습 로그로 쓰는 경우, 아래 "블로그 발행" 섹션의 블로그 주소·카테고리를 본인 것으로 바꾸면 된다.

## 절차

1. **현재 상태 파악**: `ko/study-notes.md`를 읽고 마지막 노트 번호와 "3. 질문 목록" 표의 마지막 행을 확인한다.

2. **복습 퀴즈** (노트가 4개 이상일 때): 최근 3개를 제외한 과거 노트 중 하나를 골라, "3. 질문 목록"의 **질문만** 사용자에게 보여주고 먼저 답해보게 한다. 사용자가 답한 뒤에 해당 노트의 결론을 보여주며 맞았는지 확인한다. 기억 못 했으면 "다음 파볼 주제"에 `(복습) 노트 N` 항목을 추가한다. 퀴즈는 1개만, 가볍게.

3. **이번 세션에서 배운 것 추출**: 대화를 되돌아보며 노트감을 고른다. 기준:
   - 사용자가 실제로 몰랐거나 헷갈렸던 것 (이미 아는 것 정리 X)
   - 직접 실행해서 확인한 것 (설명만 들은 것보다 우선)
   - **예상이 깨진 순간** (제일 좋은 노트감 — 실험 전 예상이 있었다면 반드시 담는다)
   - 삽질·실패도 훌륭한 노트감 (원인을 파고든 과정 포함)
   - 한 세션에서 보통 1~3개. 억지로 채우지 않는다.

4. **노트 작성** — "4. 노트" 섹션의 끝(= "5. 한눈에 보는 큰 그림" 앞)에 아래 템플릿으로 append:

   ~~~markdown
   ### 노트 N — 깨달음 한 줄 제목 (YYYY-MM-DD)

   **상황**: 무엇을 하다가 왜 걸렸나 (1~2줄)

   **예상**: 실험 전 내 예상 (있을 때만 — 깨진 예상이면 ⚠️와 연결)

   **확인한 것**:
   ```bash
   실제 실행한 명령과 결과
   ```
   설명 문단 (필요한 만큼)

   **결론**: 한 줄 요약

   **써먹기**: 실무 어디에 적용할 수 있나 한 줄 (있을 때만)

   ⚠️ 헷갈렸던 지점 (있을 때만, 없으면 생략)
   ~~~

   - 제목은 "무엇을 안다"가 아니라 **깨달음 문장형**으로 (예: "git 인증 정보는 세 군데에 흩어질 수 있다")
   - 사용자가 실제로 던졌던 질문의 말투를 살린다
   - 참고한 외부 자료(원문 트윗, 문서)가 있으면 본문에 링크로 남긴다
   - **예상**·**써먹기**는 선택 필드 — 억지로 채우지 않는다 (노트 하나 10분 리듬 유지)

5. **질문 목록 갱신**: "3. 질문 목록" 표에 행 추가 — `| # | 실제 질문 | 노트 N |`

6. **미해결 질문 적립**: 세션 중에 나왔지만 파지 못한 질문("이건 다음에 보자" 한 것)이 있으면 "6. 다음 파볼 주제"에 추가한다. 이번에 다룬 항목은 체크(`[x]`)하고 노트 번호를 단다.

7. **회고 트리거 확인**: 노트 총 개수를 세서 15의 배수를 새로 넘겼으면 정리 세션을 제안한다. 정리 세션에서는:
   - "2. 깨달은 것"을 3~5줄로 재작성
   - 맨 아래에 "예전의 나는 몰랐던 것: ..." 한 줄 회고를 남긴다 (Karpathy 원칙 3)
   - "5. 한눈에 보는 큰 그림"을 갱신

8. **자기 설명 점검 (Feynman)**: 발행 제안 전에 노트를 다시 읽고 "이 노트만 읽은 사람이 같은 상황에서 따라 할 수 있나?"를 점검한다. 빠진 전제(어떤 파일에서, 어떤 상태에서, 어떤 도구 버전에서)가 있으면 사용자에게 보충을 제안한다.

9. **블로그 발행 제안**: 노트를 적립하면 **같은 내용을 블로그에도** 발행할지 제안한다 (그 세션의 노트 1~3개를 한 글로). 발행은 반드시 사용자 확인 후에만. 절차는 아래 "블로그 발행" 참고.

10. **커밋·푸시**: 이 저장소 루트에서
    ```bash
    git add -A && git commit -m "노트 N: <제목>" && git push
    ```

## 질문 방식 (Karpathy식)

노트를 만들 때뿐 아니라 **학습 세션 중의 대화 자체**에 적용한다. Claude는 이 규칙에 따라 사용자를 유도한다.

1. **묻기 전에 예상부터**: 사용자가 질문하면 바로 답하지 말고, 짧게 "먼저 어떻게 될 것 같아요?"라고 예상을 물어본 뒤 답한다. 예상과 실제의 간극이 학습이다. (단, 사용자가 급하게 일하는 중이면 생략 — 학습 세션일 때만)
2. **답 복붙 금지, 재진술 채택**: 노트에는 AI의 설명 문장을 그대로 옮기지 않는다. 사용자가 자기 말로 다시 표현한 것(또는 사용자 말투를 살린 요약)을 쓴다. 자기 말로 못 하면 아직 이해 못 한 것.
3. **개념으로 끝내지 않기**: 모든 질문은 "직접 확인하는 최소 실험"으로 마무리한다. 실험 없이 설명만 들은 것은 노트감이 아니다.
4. **꼬리 질문 장려**: 답변 중 이해 안 되는 단어·개념이 나오면 그 자리에서 파고든다. 꼬리 질문의 흐름 자체를 질문 목록에 남긴다.
5. **컴파운딩**: 새 질문이 오면 먼저 기존 질문 목록·노트에 답이 있는지 확인하고, 있으면 그 노트를 보여준다(재질문 = 복습 기회). 관련 노트끼리는 본문에 "관련: 노트 N"으로 링크한다.
6. **원문 우선**: 요약 영상·블로그 훑기로 "배운 느낌"이 든 것은 노트감이 아니다. 공식 문서·원논문·원트윗 등 1차 소스를 확인한 것만 적는다. 노트에는 그 원문 링크를 남긴다.

참고: [전문가 트윗](https://x.com/karpathy/status/1325154823856033793), [학습의 숏츠화 트윗](https://x.com/karpathy/status/1756380066580455557)

## 정정 노트 규칙

과거 노트의 이해가 틀렸다는 걸 알게 되면, 원본은 절대 고치지 않고 **새 노트로 정정**한다:

```markdown
### 노트 M — (정정: 노트 N) 바로잡은 깨달음 한 줄 (YYYY-MM-DD)

**상황**: 노트 N에서 "..."라고 이해했는데, 어떤 계기로 틀렸다는 걸 알게 됐나

**확인한 것**: (동일 템플릿)

**결론**: 무엇이 맞는 이해인가 한 줄
```

질문 목록에도 `(정정: 노트 N)` 표기로 행을 추가한다. 이해가 바뀐 궤적 자체가 "과거의 나와 비교"의 재료다.

## 블로그 발행 (티스토리)

- 블로그: `mutpp.tistory.com` — **클론해서 쓰는 경우 본인 블로그로 교체**. Tistory Open API는 종료됨(2024-02) — **Claude in Chrome 브라우저 조작으로만 발행 가능**. 브라우저가 연결되어 있지 않으면 사용자에게 연결을 요청한다.
- **글 제목**: `[AI 학습로그] 깨달음 한 줄` 형식. **카테고리**: AI 하위 `Claude`.
- **글 구성**: 깃허브 노트와 **같은 내용**을 올린다 — 도입 1~2줄(어떤 상황이었는지) + 노트 본문 그대로(템플릿 구조 유지) + 마지막에 전체 노트 링크. 재서술하지 않는다.
- **발행본 보관**: 발행이 끝나면 그 글의 마크다운을 `blog/YYYY-MM-DD-제목슬러그.md`로 저장한다. 파일 맨 위에 발행 URL을 적는다. 노트 커밋과 함께 푸시.
- **절차와 주의점**:
  1. 글쓰기 페이지: `<블로그>/manage/newpost`
  2. 편집기를 **마크다운 모드**로 전환 (우상단 모드 드롭다운). ⚠️ 전환 시 네이티브 confirm 창이 떠서 브라우저 자동화가 멈춘다 — **사용자에게 직접 확인 버튼을 눌러달라고 요청**할 것.
  3. 제목 입력 → 본문 마크다운 입력 (자동 저장됨)
  4. "완료" 버튼 → 발행 창에서 **공개** 선택 → 사용자에게 최종 확인 후 "공개 발행" 클릭. 기본값이 비공개이므로 반드시 공개로 바꿨는지 확인.
  5. 발행 후 글 관리(`/manage/posts/`)에서 카테고리 확인, 아니면 체크박스 → 변경 → 카테고리 변경으로 이동.

## 금지 사항

- 과거 노트를 수정·삭제하지 않는다 (append-only). 오타 수정도 하지 않는다 — 틀린 이해였다면 정정 노트로 남긴다.
- 노트 본문에 이 스킬이나 세션 메타 정보(모델명 등)를 적지 않는다. 내용만 남긴다.
- 블로그 발행 버튼은 사용자 확인 없이 절대 누르지 않는다.
- 복습 퀴즈에서 사용자가 답하기 전에 정답을 먼저 보여주지 않는다.
